# Kwork → Gmail → Telegram: мониторинг заказов

## Context

Владелец бота (Рустам) — фрилансер на Kwork, специализация: лендинги и сайты под ключ для B2B и малого бизнеса (адаптивная вёрстка, быстрая загрузка, интеграция с Яндекс.Метрикой и Telegram).

Изначальный запрос — "зайди на Kwork и найди подходящие проекты" — натолкнулся на то, что биржа заказов (`kwork.ru/projects`) рендерится на клиенте (SPA), а автоматизированный вход и скрапинг через Puppeteer с пользовательской сессией нарушает пользовательское соглашение Kwork (риск блокировки аккаунта). Решение: использовать встроенные в Kwork **email-уведомления о новых заказах** (уже включены владельцем в личном кабинете) — это официальный, разрешённый канал, не требующий автоматизации самого сайта.

Письма с почты, привязанной к Kwork (Mail.ru), уже переадресуются на Gmail владельца (`trangvi801@gmail.com`) — фильтр настроен и подтверждён.

Цель этой фичи: научить сам JARVIS-бот (а не сессию Claude) самостоятельно и постоянно проверять Gmail, парсить уведомления Kwork, отбирать подходящие по профилю фрилансера через Claude, и присылать владельцу подборку в Telegram — без участия Claude Code в моменте.

## Архитектурные решения

- **Доступ к Gmail: IMAP + пароль приложения** (не Gmail API/OAuth2). Выбрано пользователем: установка занимает 2 минуты (включить 2FA → создать пароль приложения), не требует Google Cloud Console/OAuth consent screen, токены не истекают. Библиотека: `imapflow`.
- **Частота проверки: раз в час**, через `node-cron` (`0 * * * *`, `timezone: 'Europe/Moscow'`) — пакет уже в зависимостях, но в `index.js` пока не используется. Паттерн повторяет ранее удалённый модуль мониторинга рейсов (`git show 86651bf`): `cron.schedule(...)`, уведомление через `bot.sendMessage(CHAT_ID, ...)`, лог результата прогона для команды статуса.
- **Хранение состояния** (processed message-id's, время последней проверки) — локальный JSON-файл `memory_db/kwork_state.json`, по аналогии с `memory_db/flights.json` из удалённого модуля. Это техническое состояние мониторинга, а не история диалога — правило CLAUDE.md "Supabase — основное хранилище... истории" касается переписки с пользователем и не нарушается.
- **Вызов Claude для ранжирования заказов** — новый маленький модуль `claude_client.js`, повторяющий уже работающий в `index.js` (`askClaude`, строки ~32-48) fetch-вызов к `https://apinet.cloud/v1/messages` (прокси, т.к. прямой Anthropic API не используется в проекте). `askClaude` в `index.js` будет переведён на этот общий хелпер вместо дублирования — небольшой безопасный рефактор (только перенос fetch-вызова, остальная логика истории/Supabase не трогается).
- **Уведомление владельца**: через `process.env.TELEGRAM_CHAT_ID` (по аналогии с удалённым модулем рейсов). Значение нужно получить — у владельца единственный личный чат с ботом, чат-id можно достать из таблицы `messages` в Supabase (там уже есть его `chat_id` из прошлой переписки) на этапе реализации, без необходимости спрашивать пользователя.
- **Ручная проверка**: команда `/kwork_check` — запускает тот же цикл проверки немедленно (аналог `/fly_status` из истории), чтобы не ждать часовой cron при тестировании.
- **Парсер письма** (`kwork_parser.js`) пишется по образцу типичного письма Kwork (тема/ссылка на `kwork.ru/projects/...`/сумма в ₽), но т.к. реального письма пока не было, после первого реального уведомления потребуется быстрая калибровка регулярок — это отдельный чек-поинт, не блокирует остальную реализацию.

## Задачи (вертикальные срезы, по порядку зависимостей)

### 1. IMAP-доступ к Gmail + базовый ридер писем
- Добавить зависимость `imapflow`.
- `.env`: `GMAIL_USER`, `GMAIL_APP_PASSWORD` (владелец создаёт пароль приложения в настройках безопасности Google и присылает значение для `.env` — это его ручное действие вне кода).
- Модуль `kwork_mail.js`: функция `fetchUnseenKworkEmails()` — подключается к `imap.gmail.com`, ищет непрочитанные письма от `kwork.ru`/`kwork.com` в INBOX, возвращает массив `{ uid, messageId, subject, date, text, html }`; функция `markSeen(uid)`.
- **Критерий приёмки**: ручной запуск скрипта против реального аккаунта (после того как у владельца будет пароль приложения) находит входящие тестовые письма (если есть) без ошибок аутентификации; юнит-тест на построение IMAP search-критериев (чистая функция, без сети).
- **Зависимости**: нет (первая задача).

### 2. Парсер письма Kwork
- Модуль `kwork_parser.js`: функция `parseKworkEmail({ subject, text, html })` → `{ title, budget, description, link, category }`. Лучшее на что можно опереться без реального образца — структура: заголовок из subject, ссылка `https://kwork.ru/projects/...` через регулярку по `html`/`text`, сумма по паттерну `\d[\d\s]*\s?₽`, остальное — очищенный текст письма как `description`.
- Юнит-тесты (`node:test`) на 2-3 синтетических примера письма (HTML и plain-text варианты).
- **Критерий приёмки**: тесты проходят на синтетических примерах; отдельный чек-поинт после задачи 5 — прогнать на первом реальном письме и поправить регулярки при необходимости.
- **Зависимости**: нет (можно делать параллельно с задачей 1).

### 3. Общий клиент Claude (рефакторинг + переранжирование заказов)
- Новый модуль `claude_client.js`: функция `callClaude({ system, messages })`, повторяющая текущий fetch к `https://apinet.cloud/v1/messages` из `index.js`.
- `index.js`: `askClaude` переключается на этот хелпер (без изменения поведения — те же история/Supabase).
- Модуль `kwork_rank.js`: функция `rankKworkOrders(orders, profileText)` — формирует промпт с профилем фрилансера (лендинги/сайты, B2B, малый бизнес, адаптивная вёрстка, Яндекс.Метрика, Telegram) и просит Claude вернуть JSON-массив подходящих заказов с полем `reason`; фильтрует неподходящие.
- **Критерий приёмки**: регрессионный прогон существующих текстовых команд бота (`/start`, обычное сообщение) — поведение не изменилось; юнит-тест на `kwork_rank.js`, мокающий `callClaude`, проверяющий парсинг ответа модели и устойчивость к невалидному JSON.
- **Зависимости**: нет по коду, но логически нужна задача 2 (чтобы было что ранжировать) для сквозной проверки.

### 4. Состояние мониторинга (анти-дублирование)
- Модуль `kwork_state.js`: `loadState()/saveState()` для `memory_db/kwork_state.json` — `{ lastRun, processedMessageIds: [] }`.
- **Критерий приёмки**: юнит-тест на чтение/запись/инициализацию файла, если его нет.
- **Зависимости**: нет.

### 5. Сборка пайплайна + cron + Telegram + команда `/kwork_check`
- В `index.js`: функция `checkKworkOrders(notifyChatId)` — `fetchUnseenKworkEmails` → отфильтровать уже обработанные по `processedMessageIds` → `parseKworkEmail` → `rankKworkOrders` → отправить подходящие через `bot.sendMessage` → `markSeen` + обновить `kwork_state.json`.
- `cron.schedule('0 * * * *', () => checkKworkOrders(process.env.TELEGRAM_CHAT_ID), { timezone: 'Europe/Moscow' })`, активируется только если `TELEGRAM_CHAT_ID` задан (как в прошлом модуле).
- Команда `/kwork_check` — ручной запуск `checkKworkOrders(chatId)` с обратной связью в чат ("Проверяю почту...", потом результат или "Новых подходящих заказов нет").
- `.env`: добавить `TELEGRAM_CHAT_ID` (значение достанем из Supabase `messages` на этапе реализации).
- **Критерий приёмки**: сквозной ручной прогон `/kwork_check` — без реального письма Kwork должен вернуть "новых заказов нет" без ошибок; после реального письма — присылает разобранную и отранжированную карточку заказа в Telegram.
- **Зависимости**: 1, 2, 3, 4.

## Чек-поинты
- После задачи 1: подтвердить, что IMAP-логин с паролем приложения проходит (нужен пароль приложения от владельца).
- После задачи 5: дождаться первого реального письма от Kwork и при необходимости скорректировать `kwork_parser.js` под реальный формат.

## Верификация
- `node --test` — юнит-тесты парсера, ранжирования, состояния.
- Ручной прогон `node index.js` + команда `/kwork_check` в Telegram.
- Проверка, что существующие команды (`/start`, `/help`, `/clear`, обычные сообщения, голосовые) не сломались после рефакторинга `askClaude`.
